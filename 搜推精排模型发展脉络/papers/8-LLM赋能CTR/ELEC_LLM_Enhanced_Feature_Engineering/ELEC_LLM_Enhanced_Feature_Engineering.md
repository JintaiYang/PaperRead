## 一、研究背景与动机

### 1.1 领域现状

2025 年，LLM 拥有强大的文本语义理解能力，传统 CTR 模型擅长从表格数据中捕获协同信号（collaborative signals）。如何将两者的优势结合起来，同时满足在线广告系统对低延迟推理的严格要求，成为一个重要的研究问题。

### 1.2 现有方法的局限性

**语义理解与协同信号的矛盾**：传统 CTR 模型对文本语义的理解能力极弱——广告标题、商品描述等丰富的文本信息通常被简单地 hash 为 ID 特征，语义信息大量丢失。LLM 擅长理解文本含义，但在捕获协同信号方面存在天然短板——ID 特征对 LLM 来说是无意义的 token。此外 LLM 的推理延迟（通常在秒级）远不能满足在线广告系统的毫秒级要求。

**已有方案的不足**：将 LLM 作为特征提取器只利用了浅层特征且与 CTR 任务不对齐；端到端微调 LLM 做 CTR 预测推理成本太高无法在线部署。

### 1.3 本文解决方案概述

ELEC 提出"训练时用 LLM，推理时不用 LLM"的方案：通过伪孪生网络将 LLM 的语义表示注入传统 CTR 模型构建增益网络，再通过分数级和表示级双重知识蒸馏将语义理解能力迁移到仅使用表格数据的基础网络。

## 二、解决方案

### 2.1 核心思想

ELEC 的核心设计是将 LLM 的语义理解能力"固化"到轻量 CTR 模型中——训练阶段充分利用 LLM 的语义能力，推理阶段完全不依赖 LLM，从而解决 LLM 推理延迟与在线系统实时性要求之间的矛盾。

### 2.2 整体架构

#### 各模块详细说明

**模块1：LLM 适配——从语言模型到 CTR 预测器**

将表格数据转换为自然语言文本（如 "Gender is female"、"Ad title is 清爽不油腻控油洗面奶"），所有语句拼接后作为 LLM 的输入。在 LLM 顶层添加转换层（Transformation Layer）将隐状态映射为固定维度的表示向量 $\mathbf{h}_{LLM}$，再添加预测层输出 CTR 预测值。冻结 LLM 大部分参数，仅训练新增层（以及可能的 LoRA 参数），在 CTR 数据上微调。

**模块2：伪孪生网络**

- **增益网络（Gain Network）**：将 LLM 生成的高层表示向量 $\mathbf{h}_{LLM}$ 注入传统协同 CTR 模型中（拼接或交叉注意力融合），同时接收 LLM 语义表示和 CTR 模型的协同信号表示
- **基础网络（Vanilla Network）**：与增益网络结构完全相同的 CTR 模型，但不接收 LLM 的表示——仅以传统表格特征为输入。基础网络是最终在线部署的模型

两个网络形成"伪孪生"结构：共享相同的 CTR 模型架构，但输入不同。

**模块3：双重知识蒸馏**

- **分数级蒸馏（Score-Level Distillation）**：用增益网络的预测分数 $\hat{y}_{gain}$ 作为软标签，基础网络的损失包含与真实标签的交叉熵损失和与增益网络预测分数的 KL 散度损失
- **表示级蒸馏（Representation-Level Distillation）**：让基础网络的中间层表示尽量接近增益网络的中间层表示（通过 MSE 或余弦相似度损失），将"带语义的表示空间"迁移给基础网络

**模块4：模型无关性**

ELEC 框架中的 LLM 和 CTR 模型都是可替换的模块。论文验证了与 LLaMA-7B、Qwen-7B 等 LLM 的组合，以及与 DeepFM、DCN V2、AutoInt 等 CTR 模型的组合，均能取得一致的增益。

## 三、实验结果

### 3.1 数据集

| 数据集 | 规模 | 用途 |
|--------|------|------|
| Criteo | 4500万 | 公开基准 |
| 阿里内部数据集 | 大规模 | 工业验证 |

### 3.2 实验设置

#### 3.2.1 基线方法

- 传统 CTR 模型（DeepFM、DCN V2、AutoInt）
- 纯 LLM 方案
- 其他 LLM+CTR 融合方法

#### 3.2.2 评估指标

- **AUC**
- **LogLoss**

### 3.3 实验结果与分析

增益网络相比纯 CTR 模型有显著提升，证明了 LLM 语义信息的价值。经过双重蒸馏后的基础网络性能接近增益网络，远优于纯 CTR 模型——证明了蒸馏策略的有效性。基础网络的推理延迟与纯 CTR 模型相同。

### 消融实验

分数级蒸馏和表示级蒸馏各自都有贡献，两者结合效果最好。表示级蒸馏的贡献略大于分数级蒸馏，说明中间层的语义表示迁移比最终预测分数的模仿更有价值。在不同 LLM 和不同 CTR 模型的组合下，ELEC 均表现稳定。

## 四、未来工作建议

### 4.1 作者建议的未来工作

论文建议探索更高效的 LLM 适配方法、更精细的蒸馏策略、以及在更多广告场景中的验证。

### 4.2 基于分析的未来方向

1. **方向1：多轮迭代蒸馏**
   - 动机：单轮蒸馏的信息传递存在损失
   - 可能的方法：多轮交替训练增益网络和基础网络
   - 预期成果：缩小基础网络与增益网络的性能差距

2. **方向2：与生成式推荐的结合**
   - 动机：ELEC 关注单点 CTR 预测，未利用序列信息
   - 可能的方法：将 LLM 语义表示注入 HSTU 等序列模型
   - 预期成果：同时利用语义理解和序列建模能力

### 4.3 改进建议

1. **改进1：在线增量蒸馏**
   - 当前问题：蒸馏是离线一次性完成的
   - 改进方案：随着新数据到来持续更新蒸馏
   - 预期效果：基础网络能持续获得最新的语义知识

## 五、我的综合评价

### 5.1 价值评分

#### 5.1.1 总体评分
**7.0/10** - ELEC 提出了一种实用的"训练时用 LLM、推理时不用 LLM"方案，思路简洁有效，但技术难度和创新深度有限。

#### 5.1.2 分项评分

| 评分维度 | 分数 | 评分理由 |
|----------|------|----------|
| 创新性 | 7/10 | 伪孪生网络 + 双重蒸馏的思路简洁有效，但知识蒸馏本身不是新技术 |
| 技术质量 | 7/10 | 技术方案清晰完整，但整体技术难度不高，缺少深层分析 |
| 实验充分性 | 7/10 | 多种 LLM/CTR 组合的消融实验较充分，但缺少在线 A/B 测试 |
| 写作质量 | 8/10 | 论文结构清晰，动机阐述充分，方法描述简洁明了 |
| 实用性 | 7/10 | 提供了实用的 LLM 赋能 CTR 方案，但缺乏大规模在线验证 |

### 5.2 重点关注

#### 5.2.1 值得关注的技术点

- "训练时用 LLM，推理时不用 LLM"的设计哲学——解决了 LLM 推理延迟的核心矛盾
- 表示级蒸馏比分数级蒸馏更有效——中间层语义表示的迁移价值更大
- 模型无关性——可作为通用的"LLM 赋能插件"应用于任何现有 CTR 模型

#### 5.2.2 需要深入理解的部分

- 蒸馏过程中语义信息的损失有多大？基础网络真的学到了语义理解能力吗？
- 对于哪些类型的特征/场景，LLM 的语义增益最大？
- 与直接使用 LLM Embedding 作为特征的简单方案相比，ELEC 的优势有多大？
