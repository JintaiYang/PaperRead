## 一、研究背景与动机

### 1.1 领域现状

2020 年前后，DIN 和 DIEN 在用户兴趣建模上取得了显著进展，但面临一个共同瓶颈：**行为序列长度受限**。DIN 的 attention 计算复杂度与序列长度线性相关，DIEN 的 GRU 必须逐步计算，实际工业部署中行为序列通常被截断到几百条以内。然而，用户在电商平台上可能积累了数年、数万条行为记录。

### 1.2 现有方法的局限性

- **DIN/DIEN**：序列长度受限于几百条，无法利用终身行为数据
- **MIMN**（Memory-based Interest Modeling Network）：通过记忆网络将序列长度扩展到 1000 级别，但固定大小的记忆矩阵在万级序列上难以精确捕捉面向特定候选 item 的兴趣
- **核心矛盾**：更长的序列包含更丰富的兴趣信号，但计算复杂度使得直接建模不可行

### 1.3 本文解决方案概述

SIM 将超长行为序列的兴趣建模类比为**信息检索**问题，提出两阶段级联检索范式：GSU（General Search Unit）以次线性时间从超长序列中快速筛选 Top-K 相关行为，ESU（Exact Search Unit）在短子序列上精确建模兴趣。这将行为序列长度上限从 1000 推进到 54000。

## 二、解决方案

### 2.1 核心思想

SIM 的核心洞察是将超长行为序列的兴趣建模类比为信息检索中"先召回后排序"的两阶段架构：面对海量的用户历史行为"文档库"，给定一个候选广告"查询"，先用高效的粗粒度检索快速缩小范围，再用精细的模型在小范围内精确建模。

### 2.2 整体架构

![[SIM_fig1_page3.png|800]]

> 图1：SIM 的整体架构。左侧为 General Search Unit（GSU），从超长行为序列中检索 Top-K 相关行为形成子序列 SBS；右侧为 Exact Search Unit（ESU），在 SBS 上应用精确的 attention 建模。GSU 有 Hard-search 和 Soft-search 两种实现。

整体框架由两个级联的搜索单元组成：

- **GSU**：从原始超长行为序列（长度 $L$，可达 54000）中检索 Top-K 相关行为，$K$ 通常为几十到几百
- **ESU**：在短子序列 SBS 上使用 multi-head target attention 精确建模用户兴趣

#### 各模块详细说明

**模块1：General Search Unit — Hard-search（硬搜索）**

- **功能**：基于规则的无参数快速检索
- **方法**：利用 item 的类目（category）进行精确匹配
- **工业实现**：**User Behavior Tree（UBT）**，以 User ID 为一级索引、类目 ID 为二级索引
- **优势**：计算成本几乎为零、延迟极低、实现简单
- **劣势**：粒度较粗，可能遗漏跨类目关联

![[SIM_fig2_page4.png|800]]

> 图2：阿里巴巴实时预测系统（RTP）架构。SIM 通过 User Behavior Tree 索引结构实现毫秒级的行为检索。

**模块2：General Search Unit — Soft-search（软搜索）**

- **功能**：基于学习的参数化检索
- **方法**：为候选 item 和历史行为分别学习 Embedding，通过向量内积计算相关性

$$r_i = \text{sign}(\mathbf{W}_a \cdot \mathbf{e}_a, \mathbf{W}_b \cdot \mathbf{e}_{b_i})$$

- **优势**：能捕捉跨类目关联
- **劣势**：需要额外计算和存储开销
- **工业实现**：Embedding 离线预计算，通过 ANN 检索在线查询

**模块3：Exact Search Unit（ESU）**

- **功能**：在短子序列上精确建模用户兴趣
- **输入**：GSU 筛选后的子行为序列 SBS（长度 K）
- **输出**：用户兴趣表示向量
- **关键设计**：同时编码行为发生的**时间信息**（time encoding），将行为时间与当前时刻的时间差离散化后编入 Embedding

![[SIM_fig3_page5.png|800]]

> 图3：SIM 在阿里巴巴 CTR 预测系统中的完整部署架构。展示了 UBT 索引、在线检索、ESU 精确建模的完整流程。

### 2.3 训练策略

SIM 的训练包含两部分损失：主任务损失（标准 CTR 交叉熵）端到端训练整个模型；辅助任务损失用于训练 Soft-search 的 Embedding 参数。两个损失联合优化确保 GSU 和 ESU 的协同效果。

## 三、实验结果

### 3.1 数据集

| 数据集 | 规模 | 最长序列 | 数据类型 |
|--------|------|----------|----------|
| Amazon Books | 用户图书评论 | ~100 | 公开数据集 |
| Alibaba | ~5700万样本 | 54000 | 工业数据集 |

### 3.2 实验设置

#### 3.2.1 基线方法

- DIN：target-aware attention
- DIEN：GRU + AUGRU
- MIMN：记忆网络，序列长度上限 ~1000
- Avg-Pooling Long：对长序列做平均池化

#### 3.3.2 评估指标

- **AUC**：离线 ROC-AUC
- **在线 CTR/RPM**：线上 A/B 实验

### 3.3 实验结果与分析

SIM 在所有数据集上均达到最优结果。在阿里巴巴工业数据集上，SIM 将行为序列建模长度从 MIMN 的 1000 提升到 54000，AUC 提升幅度随序列长度增加而增大。

#### 结果分析

- **Hard-search vs Soft-search**：Soft-search 效果略优（能捕捉跨类目关联），但阿里最终选择 Hard-search 作为主方案（实现简单、延迟更低、效果差距不大）
- **序列长度影响**：随着序列长度从 50 增加到 54000，SIM 的 AUC 持续提升且无饱和迹象

![[SIM_fig4_page7.png|800]]

> 图4：不同模型在不同行为序列长度下的效果对比。SIM 能有效利用超长序列信息，而 DIN/DIEN 在序列超过几百后增益放缓。

### 消融实验

#### 消融结果和分析

- **去除时间编码**：效果下降，验证了行为新鲜度信息的重要性
- **GSU 方案对比**：Soft-search > Hard-search > Random-search，验证了检索质量对最终效果的影响
- **K 值影响**：K 从 10 增加到 200 时效果持续提升，之后趋于饱和

### 实验结果图

![[SIM_fig5_page7.png|800]]

> 图5：系统性能分析。展示了不同行为序列长度下的推理延迟和吞吐量，验证了 SIM 在超长序列上的效率优势。

### 在线 A/B 测试

SIM 在阿里巴巴展示广告系统的在线 A/B 测试中取得了 **CTR 提升 7.1%、RPM 提升 4.4%** 的效果，已服务阿里主要广告流量。

## 四、未来工作建议

### 4.1 作者建议的未来工作

论文在结论中提到，未来的研究方向包括探索更精细的 GSU 检索方法、以及将 SIM 的思想扩展到其他推荐场景。

### 4.2 基于分析的未来方向

1. **方向1：端到端的长序列建模**
   - 动机：SIM 的两阶段架构中 GSU 和 ESU 的优化目标不完全一致
   - 可能的方法：设计 O(L) 以下复杂度的端到端 attention（如 ETA 的 SimHash）
   - 预期成果：消除两阶段的信息损失
   - 挑战：在保持效率的同时实现端到端优化

2. **方向2：动态行为索引更新**
   - 动机：UBT 索引的更新存在延迟，可能遗漏最新行为
   - 可能的方法：实时流处理 + 增量索引更新
   - 预期成果：更及时地利用最新行为信息
   - 挑战：实时更新的系统复杂度

### 4.3 改进建议

1. **改进1：多粒度检索**
   - 当前问题：Hard-search 只用类目匹配，粒度单一
   - 改进方案：同时在品类、品牌、价格带等多个维度检索
   - 预期效果：更全面的相关行为召回

### 5.2 重点关注

#### 5.2.1 值得关注的技术点

- "先检索后建模"的范式——将推荐系统的级联架构引入用户行为序列建模
- User Behavior Tree（UBT）索引结构——工业级超长序列的高效存储和检索方案
- 时间编码的设计——行为新鲜度信息对兴趣建模的重要性

#### 5.2.2 需要深入理解的部分

- Hard-search vs Soft-search 的工业选择——为什么阿里最终选择了更简单的 Hard-search？

- 序列长度无饱和现象——为什么更长的序列总能带来提升？
